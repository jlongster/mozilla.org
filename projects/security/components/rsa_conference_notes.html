Notes from RSA Security Conference 2/18-2/22 2002

Michael Howard, Microsoft's Secure Windows Initiative

Microsoft decided they need to learn from past bugs. Close the loop between threat identification -> product planning -> product design -> find vulnerability -> write patch. The last necessary step is to evaluate the cause of the mistake that led to the vulnerability and include that in planning and threat assessment. This includes reviewing past security bugs to look for patterns. Also identified a need for better developer education and threat models.

All new engineers will now go through training on writing secure code (secure code != security code). Not about how crypto works, but how to use it.

Design Issues

  • Too many features were enabled by default. Sysadmins don't know what they can safely turn off, so they tend to leave everything on.
  • Apply the principle of least privilege - make sure programs run with as few privileges as possible. Don't test in admin (root) mode.

Other Techniques

  • Get the execs in the game. To do this, you need to "put the numbers in security" by showing potential damages, return-on-investment.
  • A checklist of common problems to avoid helps, but isn't sufficient.
  • need to teach developers how to identify issues
  • Developers seem excited by the opportunity to learn about writing secure code - new thing to learn.
  • MS will spend February doing nothing but security - like kicking a bad habit.
  • Developers will be held accountable for holes in their code (how?)
  • They have assigned security-review bugs against each module in the code. We could do this, but we would need a well-defined deliverable.
  • PM's should justify all defaults - if 90% of users don't use a feature, it should be off by default.
  • Techniques for finding buffer overruns:
    • Follow data into the system, and question assumptions about the format of the data at various points
    • Look for common "dangerous API" patterns
    • Byte count vs. word count (two-byte strings)
    • Build scanning tools into precheckin tests (or tinderbox, maybe?)
    • Always specify buffer size in function call, don't assume (if you assume a null-terminated string, can you be sure?)
    • Use memset to verify buffer size

Doc Shankar, IBM, Linux Security

Open source does not guarantee secure software. Need to generate interest from security experts in reviewing your product, and train community members to spot problems. Many participants will not notice a buffer overrun when they see one. Linux has received good scrutiny.

Benefits

  • Programmers will be more cautious if their work is open to scrutiny (?!?)
  • Effective peer review/audit
  • Faster patches - how can Netscape leverage this?
  • Security researchers may be interested in participating
  • Open-source crypto is a must, unless you're the NSA.

Bruce Schneier, Counterpane

  • Security is a key enabling technology for the Internet - lack of security inhibits many Internet activities.
  • Security is a business problem and a human problem, not a technology problem. Processes, not technology, protect us from crime in the real world.
  • Software makers have been using the military model of threat avoidance - try to build an impenetrable fortress. Emphasis on protection, closing the holes with technology. This has limits - no fortress is impenetrable.
  • We should be using a business model - risk management.
    • It's OK to fail occasionally
    • Use technology, processes, and insurance to mitigate risk
  • Insurance turns a variable risk into a fixed cost; businesses like this.
  • In the future, insurance companies will insure networks against hacking and define standards for securing those networks, analogous to standards for fire safety, which are also set by insurers.
  • Key military strengths: quick reaction to attack, and superior knowledge of the territory. Both can be applied to computers.

Types of Security Problems

  • Viruses / Worms / Trojans
  • Buggy Code
  • Bad Data:
    • Buffer Overflows
    • Parsing Errors
    • Bad CGI input (or any data presented to web app backend)
    • Format string vulnerabilities
  • Misconfiguration, due to operator error or too many default services
  • Human Error:
    • Weak Passwords
    • Social Engineering
  • DoS / DDoS

Why Security is Hard

  • No demand-pull for security/privacy. Customers will say they want security and privacy, but it doesn't really affect buying decisions. Currently, there is incentive to create the lowest-quality product the market will bear.
  • With no demand pull, management won't allocate resources. Need to get the boss involved.

Recommendations

  • Complexity is the enemy of security. Turning off services or putting them on different hosts reduces complexity and possible unintended interactions
  • Risk Assessment: Risk = Threat * Vulnerability * Impact
  • Pitch it to management in terms of return on investment: risks, cost of security problems, cost of securing systems/code.
